# Story 1.1: Configurar Vitest e Testes Unitários — Brownfield Addition

## User Story

Como desenvolvedor,
Eu quero um framework de testes configurado no monorepo,
Para que eu possa escrever e executar testes automatizados com confiança.

## Story Context

**Existing System Integration:**

- **Contexto:** O projeto não possui nenhum framework de testes configurado. Não há `vitest`, `jest`, `playwright` ou qualquer outra ferramenta de teste.
- **Tecnologia:** Vitest será adicionado (já usa Vite 8.0.8 no web, compatibilidade nativa)
- **Segue padrão:** Configuração via `vitest.config.ts` em cada workspace, integrado ao Biome
- **Pontos de contato:** `package.json` (scripts), `biome.json` (ignores), todos os pacotes

## Acceptance Criteria

**Functional Requirements:**

1. Vitest está configurado como devDependency no root do monorepo (catalog)
2. Cada package (`apps/web`, `apps/server`, `packages/db`, `packages/auth`, `packages/env`, `packages/ui`) tem seu próprio `vitest.config.ts`
3. Script `test` e `test:run` adicionados ao root `package.json` via `bun run --filter '*'`
4. Pelo menos um teste de exemplo (`*.test.ts`) existe em cada package para validar o setup
5. Biome configurado para ignorar arquivos de cobertura (`coverage/`)
6. `.gitignore` atualizado com `coverage/`

**Integration Requirements:**

7. Script `test` roda via `bun run test` (watch mode opcional)
8. Script `test:run` roda uma vez (CI mode)
9. Testes existentes (se houver) continuam passando

**Quality Requirements:**

10. Configuração inclui `@vitest/coverage-v8` para relatórios de cobertura
11. Projetos compartilham config base via `packages/config/vitest.base.ts`

## Technical Notes

- **Integration Approach:** Vitest é a escolha natural pois o projeto já usa Vite 8. — compatibilidade total, mesma equipe de mantenedores, mesmo transform pipeline.
- **Existing Pattern Reference:** Seguir padrão de `tsconfig.base.json` em `packages/config/` — criar `vitest.base.ts` similar.
- **Key Constraints:**
  - Usar `bun run --filter '*'` para scripts cross-workspace
  - `@colorir/*` imports precisam ser resolvidos via `resolve.alias` nos configs de teste
  - Server tests podem precisar de mock para `process.env`

## Risk and Compatibility Check

- **Primary Risk:** `@colorir/*` imports podem não resolver corretamente com Vitest sem alias config
- **Mitigation:** Usar `resolve.alias` no `vitest.base.ts` mapeando `@colorir/*` para os paths corretos
- **Rollback:** Reverter mudanças no `package.json`, remover `vitest.config.ts` files
