# Story 1.2: Adicionar Validação Zod ao Endpoint /ai — Brownfield Addition

## User Story

Como desenvolvedor,
Eu quero validação de entrada no endpoint `POST /ai`,
Para que requisições mal formatadas sejam rejeitadas com erros claros e o servidor permaneça robusto.

## Story Context

**Existing System Integration:**

- **Integrates with:** `apps/server/src/index.ts` — endpoint `POST /ai`
- **Technology:** Zod 4.1.13 (já disponível no monorepo via catalog), Hono 4.8.2
- **Follows pattern:** `packages/env/src/` já usa Zod para validação de env vars — mesmo padrão
- **Touch points:** `apps/server/src/index.ts`, opcionalmente `packages/ai-validation/` ou inline

## Acceptance Criteria

**Functional Requirements:**

1. Schema Zod criado para o body do `POST /ai` validando:
   - `messages` — array obrigatório, não vazio
   - Cada mensagem: `{ role: "user" | "assistant", content: string }`
   - `id` opcional por mensagem
2. Validação executada antes de chamar `streamText`
3. Resposta com status `400` e mensagem de erro detalhada quando validação falha
4. Resposta com status `422` para tipos de dados inválidos

**Integration Requirements:**

5. Endpoint continua funcionando para requisições válidas
6. Streaming response preservado para requisições válidas
7. Nenhuma regressão em outros endpoints (`/api/auth/*`, `GET /`)

**Quality Requirements:**

8. Testes unitários para o schema de validação
9. Tipos TypeScript exportados do schema para uso no frontend

## Technical Notes

- **Integration Approach:** Schema Zod definido inline ou em arquivo separado em `apps/server/src/validation/ai.ts`. Reutilizar `zod@^4.1.13` já existente no monorepo.
- **Existing Pattern Reference:** `packages/env/src/server.ts` (16 lines) mostra o padrão Zod usado no projeto.
- **Key Constraints:**
  - Hono 4.x não tem integração nativa com Zod — usar validação manual com `c.req.json()` + `schema.parse()`
  - `convertToCoreMessages()` do Vercel AI SDK é chamado depois — validar antes

## Definition of Done

- [x] Schema Zod criado e exportado
- [x] Validação integrada ao handler do `POST /ai`
- [x] Erros de validação retornam status code apropriado (400/422)
- [x] Tipos exportados para consumo do frontend
- [x] Testes unitários para o schema
- [x] `bun run test` passa

## Risk and Compatibility Check

- **Primary Risk:** Mudança na estrutura de `messages` recebida pelo `convertToCoreMessages()` do AI SDK
- **Mitigation:** Validar contra a interface esperada pelo AI SDK (role: "user" | "assistant", content: string)
- **Rollback:** Reverter adição da validação no handler, manter schema para uso futuro
