# Story 1.5: Gerar Migrations Iniciais do Drizzle — Brownfield Addition

## User Story

Como desenvolvedor,
Eu quero migrations do Drizzle geradas a partir do schema atual,
Para que mudanças no banco de dados sejam versionadas e aplicadas de forma segura.

## Story Context

**Existing System Integration:**

- **Integrates with:** `packages/db/` — Drizzle ORM schema e config
- **Technology:** Drizzle ORM 0.45.1, Drizzle Kit 0.31.8
- **Follows pattern:** `drizzle.config.ts` já existe e está configurado — só não foi executado
- **Touch points:** `packages/db/drizzle.config.ts`, `packages/db/src/schema/`, `packages/db/src/migrations/` (criar)

## Acceptance Criteria

**Functional Requirements:**

1. Migrations geradas do schema atual (4 auth tables) sem erros
2. Migrations salvas em `packages/db/src/migrations/`
3. Script `db:generate` no root `package.json` adicionado/se mantido
4. Script `db:migrate` funcional
5. `README.md` do projeto atualizado com instruções para migrações (vs `db:push`)

**Integration Requirements:**

6. Schema atual (`auth.ts`) não é modificado — snapshot fiel
7. `db:push` ainda funciona como alternativa

**Quality Requirements:**

8. Migrations revisadas manualmente antes do commit
9. Rollback testado via `drizzle-kit drop` ou migration down

## Technical Notes

- **Integration Approach:** Executar `bun run db:generate` (que chama `drizzle-kit generate`).
- **Existing Pattern Reference:** `packages/db/drizzle.config.ts` já está configurado com `out: "./src/migrations"`.
- **Key Constraints:**
  - Precisa de conexão com PostgreSQL rodando para gerar migrations
  - Usar `docker compose up -d postgres` para iniciar DB local
  - Arquivo de migração gerado precisa ser commitado

### Pré-requisitos

```bash
docker compose up -d postgres   # Iniciar PostgreSQL
bun run db:generate             # Gerar migrations do schema
bun run db:migrate              # Aplicar migrations
```

## Definition of Done

- [x] `drizzle-kit generate` executado com sucesso
- [x] Migrations geradas em `packages/db/src/migrations/`
- [x] `db:migrate` aplica migrations (manual via psql — `drizzle-kit migrate` hangs com pg driver no Windows)
- [x] Migrations commitadas no repositório (c16a775)
- [x] README atualizado com workflow de migrations

## Dev Agent Record

**Agent Model Used:** Dex (dev) — YOLO mode

**Debug Log Reference:**
- `drizzle-kit generate` falhou inicialmente porque `auth.test.ts` na pasta schema importa vitest (CommonJS incompatibility)
  - Solução: mover `auth.test.ts` para `packages/db/src/__tests__/auth.test.ts`
- `drizzle-kit migrate` trava (hangs) no Windows com driver pg — não retorna erro nem sucesso
  - Migração aplicada manualmente via `psql` com `Get-Content ... | docker exec -i postgres psql`
- Drizzle-kit gera arquivo `meta/_journal.json` e `meta/0000_snapshot.json` para tracking (não usa `__drizzle_migrations` table como outras ORMs)

**Completion Notes:**
- Migração `0000_hot_diamondback.sql` gerada com sucesso — 4 tabelas (user, session, account, verification), 3 indexes, 2 FKs
- README.md atualizado com workflow de migrations e diff entre `db:push` e `db:generate` + `db:migrate`
- `db:migrate` via drizzle-kit hangs no Windows — aplicar migrations manualmente via psql como workaround
- Rollback testado: `DROP TABLE ... CASCADE` + re-aplicação via SQL direto

## Risk and Compatibility Check

- **Primary Risk:** Drizzle pode gerar migration com diferenças inesperadas se o schema não estiver perfeitamente sincronizado com o banco
- **Mitigation:** Revisar migration gerada antes de commitar; testar apply + rollback em DB local
- **Rollback:** `drizzle-kit drop` ou deletar migration file e usar `db:push`
